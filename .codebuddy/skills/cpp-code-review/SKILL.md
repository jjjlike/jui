# C++ 代码自我审查

## 概述

对自己写的代码进行七维系统性审查。作为独立开发者，你没有 PR reviewer，所以必须建立一个严格的自我审查纪律。每段代码在"完成"之前都需要通过这个审查。

**自动化策略：** Agent 自动执行审查流程，自动运行 clang-tidy、clang-format 检查、ASan 检查，自动修复发现的问题。仅在发现架构级别的需要重新设计时才中断。

**审查标准：** 当改动肯定提升了代码整体质量时就可以放行，即使不完美。完美的代码不存在——目标是持续改进。

## 何时使用

- 完成一个功能切片后
- 修复 Bug 后（审查修复和回归测试）
- 重构后
- 提交前最后的检查

## 七轴审查

### 1. 正确性

代码做到它声称要做的事情了吗？

- 是否符合规格或任务要求？
- 边界条件处理了吗（null、空、零值、极大极小值）？
- 错误路径处理了吗（不只是 happy path）？
- 是否通过了所有测试？测试本身是否正确？
- 有 off-by-one 错误吗？有悬挂指针吗？
- 多线程代码有数据竞争吗？

### 2. 可读性与简洁性

另一个开发者（或 6 个月后的你）能不用解释就看懂吗？

- 命名是否清晰且符合项目约定？（避免 `tmp`、`data`、`result` 这类无信息量命名）
- 控制流是否直接？（避免深层嵌套、goto、复杂的 lambda 嵌套）
- 代码是否逻辑分组？
- 有没有"聪明"但难懂的写法？

**C++ 专项检查：**
```cpp
// ✗ 过度"聪明"
auto result = std::accumulate(v.begin(), v.end(), 0,
    [](int a, int b) { return a ^ (b << 1); });

// ✓ 简单直接
int result = 0;
for (int x : v) {
    result ^= (x << 1);
}
```

- **能否用更少代码？** C++ 容易写成"1000行只需要100行"的风格
- **这个模板/抽象值得它的复杂度吗？** 不要为了"灵活性"而泛化
- Lifetime 是否清晰？查看所有裸指针和引用，确认它们不会悬空

### 3. 中文注释质量 ⭐（必须）

注释是否用中文编写、是否详细充分：

- [ ] 每个类/结构体有中文注释说明其职责和生命周期
- [ ] 每个公开函数有中文注释说明功能、参数含义、返回值和异常
- [ ] 每个成员变量有中文注释说明含义和约束
- [ ] 关键算法段落有中文注释解释设计决策——不仅说"是什么"，还要说"为什么"
- [ ] TODO/FIXME/HACK 用中文写了上下文（不能只写 TODO）
- [ ] 没有英文注释（除非引用标准文档中的术语）

```cpp
// ✗ 差：注释不足，后来的人完全看不懂意图
auto result = ProcessItems(items, kModeFast);

// ✗ 差：英文注释
// Process items in fast mode
auto result = ProcessItems(items, kModeFast);

// ✓ 好：中文详细注释，说清场景和理由
// 当数据量超过 10000 条时，使用快速模式跳过完整性校验。
// 快速模式仅检查主键冲突，不验证外键约束（约节省 60% 处理时间）。
// 注意：调用方需确保上游数据已通过完整性检查。
auto result = ProcessItems(items, kModeFast);
```

### 4. 中文编码安全 ⭐（必须）

- [ ] 源文件保存为 UTF-8 with BOM（MSVC 正确解析中文的必需条件）
- [ ] 用户可见的字符串使用 `std::wstring` 而非 `std::string`
- [ ] Windows API 调用使用 W 后缀版本（`CreateFileW` 而非 `CreateFile`）
- [ ] 控制台/日志输出中文前正确设置了编码（`SetConsoleOutputCP(CP_UTF8)` 或 `std::wcout`）
- [ ] 文件 I/O 中使用 `std::wstring` 路径（Windows 支持 Unicode 路径）
- [ ] JSON/XML 等数据格式中读写中文时正确指定了编码

```cpp
// ✗ 危险：窄字符串中文可能在 GBK 环境下乱码
void showError(const std::string& msg) {
    MessageBoxA(nullptr, msg.c_str(), "错误", MB_OK);
}

// ✓ 安全：宽字符串 + Unicode API，全平台无乱码
void showError(const std::wstring& msg) {
    MessageBoxW(nullptr, msg.c_str(), L"错误", MB_OK);
}
```

### 5. 架构

改动是否符合系统设计？

- 遵循已有模式还是引入新模式？新模式合理吗？
- 模块边界清晰吗（public/private 划分合理）？
- 有没有重复代码应该被共享的？
- 依赖方向正确吗（避免循环依赖）？
- 抽象层次合适吗（不过度设计，不过于耦合）？
- UI 相关：是否使用了原生 API + D2D 而非引入第三方框架？

### 6. 内存安全

这是 C++ 独有的、最重要的维度。详见 `cpp-memory-safety`。

- 有没有裸 `new`/`delete`？（应使用 `std::make_unique`/`std::make_shared`）
- 所有指针是否都有明确的所有者？
- 有没有返回局部变量的引用/指针？
- 有没有悬空指针风险（容器元素地址在插入后失效）？
- 有没有未初始化变量？
- 资源是否用 RAII 管理？
- lambda 捕获是否正确（按值 vs 按引用）？

### 7. 性能

有没有引入明显的性能问题？详见 `cpp-performance-optimization`。

- 有没有不必要的拷贝？（能用 `const&` 或 `std::move`？）
- 热路径中有没有不必要的堆分配？
- 有没有 O(N²) 算法用在可能很大的 N 上？
- 虚函数调用在热路径中是不是瓶颈？
- `std::map` vs `std::unordered_map` 选择是否合理？

## 自动审查工具链

Agent 将自动运行以下工具：

```bash
# 1. 格式化检查
clang-format --dry-run -Werror src/**/*.cpp include/**/*.h

# 2. 静态分析
clang-tidy src/**/*.cpp -- -std=c++17 -I include

# 3. 编译（所有警告为错误）
cmake --build build  # 自动带 -Wall -Wextra -Werror

# 4. 测试 + ASan
cmake --build build && ctest --test-dir build

# 5. 如果有基准测试
cmake --build build --target bench
```

完成后检查是否留下了死代码：

```

- src/core/old_init.cpp — 新的初始化路径不再使用
```

**注意：** 死代码直接删除（项目目录内删除无需确认）。如果某段代码你不确定是否仍在使用，删除前先搜索全项目引用后再删除。

## 常见借口

| 借口 | 现实 |
|------|------|
| "我自己写的，肯定对" | 作者对自己的假设是最盲目的。每段代码都需要审一遍 |
| "能用就行" | 能用的代码如果不可读、不安全、架构错误，会产生复利式技术债 |
| "以后整理" | "以后"永远不会来。审查是质量门——现在就用 |
| "AI 生成的代码应该没问题" | AI 代码需要更多审查而非更少。它自信且合理，甚至当错误时也太像那么回事 |

## 红旗信号

- 写完代码直接标记完成，跳过审查
- 审查只检查"测试过了吗"而忽略其他维度
- 有明显的 C 风格指针操作混入 C++ 代码
- 新增了不必要的第三方依赖（尤其是 UI 框架）
- 审查发现的问题被标记"以后处理"
- 公开接口没有中文注释
- 注释是英文或注释过于简略（如 `// 初始化` 对 `init()` 函数）
- 用户可见文本使用窄字符串（`std::string`/`char*`）而非宽字符串
- 源文件缺少 UTF-8 BOM（MSVC 下中文注释乱码的根源）

## 验证

- [ ] 七轴审查全部通过（至少无 ❌ 项）
- [ ] 所有公开接口有中文注释，关键算法有中文解释
- [ ] 源文件 UTF-8 with BOM，用户可见文本使用宽字符
- [ ] clang-format 检查通过
- [ ] clang-tidy 无严重警告
- [ ] 编译通过（带 -Werror）
- [ ] 测试全部通过
- [ ] ASan 无报错
- [ ] 死代码已清理
- [ ] UI 相关代码未引入第三方框架（除非已确认需要）
