# C++ 增量实现

## 概述

每次只实现一个完整的垂直切片——先让一块功能跑通，编译通过、测试通过，再继续扩展。不要试图一次实现整个功能。这是让大型功能变得可控的执行纪律。

**自动化策略：** 每完成一个切片，自动编译-测试-提交。不需要你逐一切片确认。编译失败则自动回退修正，不需要问你"要不要继续"。仅在无法自动修复时才中断。

## 何时使用

- 实现任何涉及多文件修改的功能
- 从任务拆解结果开始构建新功能
- 重构现有代码
- 任何时候想一次写超过 100 行 C++ 代码却不编译

## 增量循环

```
┌──────────────────────────────────────────────┐
│                                              │
│   实现 ──→ 编译 ──→ 测试 ──→ 验证 ──→ 提交  │
│    ▲                                  │      │
│    └────────── 下一个切片 ←───────────┘      │
│                                              │
└──────────────────────────────────────────────┘
```

每个切片：
1. **实现**——最小的完整功能片段
2. **编译**——`cmake --build build` 必须通过
3. **测试**——运行相关测试套件
4. **验证**——确认功能端到端可用
5. **提交**——自动 `git commit`（见下方提交规范）
6. **继续**——进入下一个切片

### 自动化提交规范

```bash
# Agent 自动提交格式
git add src/module/ files tests/module/ files
git commit -m "[cpp-skills] feat(模块名): 切片 N — 简短描述

- 做了什么
- 测试状态: N passed / 0 failed"
```

**提交粒度：** 每个切片一个 commit。编译失败不提交。

## 切片策略

### 策略 1：垂直切片（首选）

```cpp
// 切片 1：定义数据类 + 基础单元测试
// DataConfig 类定义、构造、getter/setter 测试
// → 编译通过，测试通过

// 切片 2：实现配置加载器 + 单元测试
// ConfigLoader::load() + ConfigLoader::save()
// → 编译通过，测试通过

// 切片 3：集成到应用入口 + 集成测试
// main() 中调用 ConfigLoader，端到端验证
// → 编译通过，测试通过
```

### 策略 2：接口优先

当多个组件并行开发时：

```cpp
// 切片 0：定义接口（纯虚类 / concept）
class ILogger {
public:
    virtual ~ILogger() = default;
    virtual void log(Level, std::string_view) = 0;
};

// 切片 1：实现 ConsoleLogger 并对接口测试
// 切片 2：实现 FileLogger 并对接口测试
// 切片 3：替换旧 Logger，全量回归测试
```

### 策略 3：风险优先

把最不确定的部分第一个做：

```cpp
// 切片 1：验证第三方库集成是否可行（高风险）
// 切片 2：在已验证的集成基础上构建功能
// 切片 3：添加边界条件处理
```

### 策略 4：UI 原生优先

当实现 UI 功能时，第一个切片必须是原生 API 的可行性验证：

```cpp
// 切片 0：用 Win32 API 创建基础窗口，验证消息循环可行
// 切片 1：在窗口上用 Direct2D 绘制基本图形
// 切片 2：封装成可复用的控件基类
// 切片 3：实现具体业务 UI 逻辑
// 不要在切片 0 就决定引入 Qt/ImGui 等第三方框架
```

## C++ 实现规则

### 规则 0：编写前先想

在写任何代码前问自己：**"有没有更简单的方法？"**

```cpp
// ✗ 过度设计：为两种格式写模板
template<typename T>
concept Serializable = requires(T t) { t.serialize(); };

// ✓ 简单直接：先实现一个具体版本
std::string serialize(const Config& config);
```

### 规则 0.3：注释必须用中文

在写代码的同时写注释，不要写完再补：

```cpp
// ✗ 无注释 / 英文注释
double calculateRatio(int a, int b) {
    return static_cast<double>(a) / b;
}

// ✓ 中文详细注释，说明业务含义和边界条件
/// 计算两个整数的比值。
/// @param a 分子，表示已完成的数量
/// @param b 分母，表示总数量
/// @return 比值（0.0 ~ 1.0），如果分母为 0 则返回 0.0
/// @note 调用者应确保 a <= b，否则返回值可能大于 1.0
double 计算完成率(int 已完成, int 总数量) {
    if (总数量 == 0) return 0.0;
    return static_cast<double>(已完成) / 总数量;
}
```

### 规则 0.5：范围纪律

只动当前任务需要的文件：

```
注意到了但不碰：
- src/utils/string_helper.cpp 有明显的性能优化空间（与本任务无关）
- include/common/types.h 可以整理 include 顺序（与本任务无关）
→ 创建一条 TODO 备注，但不在此次修改
```

### 规则 1：每次只做一件事

每个增量改变一个逻辑单元。不要一个 commit 混入新功能和重构。

### 规则 2：保持可编译

每个切片结束后，项目必须编译通过且测试通过。

### 规则 3：安全默认值

新增的 C++ 代码默认使用安全行为：

```cpp
// ✓ 安全默认：最小权限、RAII 自动管理
class FileWriter {
public:
    explicit FileWriter(const std::filesystem::path& path)
        : m_file(path, std::ios::out | std::ios::trunc) {}
    // 析构自动关闭文件
private:
    std::ofstream m_file;
};
```

### 规则 4：每个切片独立可回退

- 新增文件容易回退
- 修改已有接口要最小化且聚焦
- 不要在同一个 commit 里删除旧代码又新增替代代码

## C++ 编译时间管理

大项目编译慢是常见痛点。增量实现时的注意事项：

- 每个切片最多触发 5-10 个翻译单元重编译
- 尽量改 `.cpp` 而非 `.h` —— 头文件改动成本高
- 使用 pimpl 或前向声明隔离改动
- 如果使用预编译头（PCH），确保常用头在 PCH 中

## 常见借口

| 借口 | 现实 |
|------|------|
| "最后一起测试就行" | 错误会叠加。切片 1 的 Bug 会让切片 2-5 都错误 |
| "一次性写完更快" | 感觉更快，但一旦出错，500 行改动中找出问题要花数小时 |
| "改动太小不值得单独提交" | 小提交零成本。大提交隐藏 Bug 且难以回退 |
| "顺便重构一下这个文件" | 重构和新功能混一起，让两者都更难调试 |

## 红旗信号

- 写了超过 100 行代码还没编译
- 一个增量中混入多个无关改动
- "顺便加一下这个"——范围膨胀
- 跳过编译/测试步骤以"加快进度"
- 大量未提交的改动堆积

## 验证

每个切片完成后：

- [ ] `cmake --build build` 编译通过
- [ ] `ctest --test-dir build` 测试通过（至少相关测试）
- [ ] 不涉及本次改动的测试全部通过（无回归）
- [ ] Git 已自动提交
