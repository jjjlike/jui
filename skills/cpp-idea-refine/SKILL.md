# C++ 想法精炼

## 概述

将原始想法打磨为清晰、可执行的 C++ 实现概念。作为独立开发者，这个阶段帮你避免在错误方向上浪费数天编码。Agent 将主动推进对话，仅在关键决策点征求你的意见。

## 何时使用

- 有一个功能想法但不确定从何下手
- 需要在多个实现方案中选择
- 想先评估难度和风险再开始编码
- 想法太庞大，需要确定 MVP 范围

## 三阶段流程

### 阶段 1：展开（发散思维）

**目标：** 将原始想法展开，生成多个可行方向。

1. **用一句话重述想法** —— 明确要解决什么问题。格式："如何让用户在 [场景] 中完成 [目标]？"

2. **自动评估上下文** —— 阅读项目现有代码，了解：
   - 现有的架构模式（信号槽？回调？观察者？）
   - 已有的工具类和基础设施
   - C++ 标准版本和编译器约束
   - 依赖的第三方库（Qt、Boost、STL 可用特性）

3. **生成 5-8 个方案变体**，用以下视角审视：
   - **最简实现**：用最少代码能跑起来的版本
   - **性能优先**：极致优化的版本
   - **灵活扩展**：为未来变化预留接口的版本
   - **STL 优先**：尽量只用标准库，零外部依赖
   - **Qt/框架优先**：充分利用现有框架能力
   - **现代 C++**：用 C++17/20 特性（`std::optional`、`std::variant`、concepts）
   - **C with Classes**：回归简单，减少模板和抽象

**注意事项：**
- 不要生成 20+ 个肤浅方案，5-8 个有深度考虑的胜过一堆浅的
- 每个方案都要给出存在的理由
- 代码量的估算要实际

### 阶段 2：评估收敛

根据你的反馈，将方案收敛到 2-3 个方向，并从三个维度评估：

1. **复杂度评估**
   - 预计代码行数
   - 新引入的头文件/依赖数量
   - 对现有架构的影响范围

2. **性能评估**
   - 热路径分析：哪些操作会被频繁调用？
   - 内存分配模式：能否避免堆分配？
   - 多线程考虑：是否需要加锁？

3. **维护评估**
   - 6 个月后你还能看懂吗？
   - 如果需要改动，会影响多少地方？
   - 模板代码多了会不会让编译变慢？

**诚实评估，不做应声虫。** 如果一个方案有明显问题（过度设计、性能隐患、可维护性差），直接指出。

### 阶段 3：输出交付

生成一个 Markdown 文档保存到 `docs/ideas/[idea-name].md`（自动保存，不等待确认）：

```markdown
# [想法名称]

## 问题描述
[一句话描述要解决的问题]

## 推荐方案
[选定的方案以及理由 — 2-3 段]

## MVP 范围
- 包含：[核心功能列表]
- 不含：[明确排除的功能]

## 不做的理由
- [功能 1] — [为什么不现在做]
- [功能 2] — [为什么不现在做]

## 技术决策
- C++ 标准：C++17 / C++20
- 关键类型设计：[简要描述核心类/结构体]
- 第三方依赖：[需要的库]

## 风险评估
| 风险 | 等级 | 缓解策略 |
|------|------|----------|
| [描述] | 高/中/低 | [如何应对] |
```

## 常见借口

| 借口 | 现实 |
|------|------|
| "先写了再说" | 缺乏打磨的代码往往需要推倒重来，浪费数天时间 |
| "这个想法很简单" | 简单的想法也可能有复杂的实现。花 15 分钟评估，可能省 15 小时 |
| "我心里已经想清楚了" | 写下来才能发现逻辑漏洞。不写下来的想法只是幻想 |

## 红旗信号

- 没想清楚目标用户场景就开始写代码
- 生成了一大堆肤浅方案而不是深入思考
- 输出里没有"不做的理由"列表
- 跳过了代码库上下文分析

## 验证

- [ ] 问题描述清晰（一句话能说明白）
- [ ] 至少对比了 3 个方案
- [ ] MVP 范围明确（包含什么、不含什么）
- [ ] "不做的理由"列表存在
