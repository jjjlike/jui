# C++ 任务规划与拆解

## 概述

将规格分解为小型、可验证的任务，每个任务都有明确的验收标准。好的任务拆解是"可靠完成"和"一团乱麻"之间的区别。每个任务应该小到可以在一次专注的编码会话中实现、测试和验证。

**自动化策略：** Agent 自动拆解任务并排序，不需要逐任务确认。按拆解结果自动执行。

## 何时使用

- 有规格说明，需要拆解为可执行单元
- 任务太大太模糊，无法直接下手
- 实现顺序不明确（依赖关系复杂）

## 规划流程

### 第一步：只读分析

在写任何代码之前，先进入只读模式：
- 读取规格和相关代码区域
- 识别现有模式和约定（C++ 命名空间组织、include 约定）
- 绘制组件依赖图
- 标注风险和未知点

**不要在此阶段写代码。**

### 第二步：识别依赖图

画出依赖关系，C++ 项目的典型依赖链：

```
基础类型/结构体
    │
    ├── 工具类（Logger、ConfigManager）
    │       │
    │       └── 业务逻辑类
    │               │
    │               └── UI/交互层（如果有）
    │
    └── 测试基础设施
```

C++ 特殊考虑：
- 头文件依赖（`#include` 层级）—— 向前声明能减少编译依赖的尽早用
- 单例/全局状态—— 标注哪些类有全局状态，它们必须优先处理
- 跨模块调用—— 标注纯虚接口（`I*` 类）和具体实现之间的依赖

### 第三步：垂直切片

构建完整的垂直切片，而非水平层：

**差（水平分层）：**
```
任务 1：构建所有数据类
任务 2：实现所有业务逻辑
任务 3：实现所有 UI
任务 4：对接一切
```

**好（垂直切片）：**
```
任务 1：用户配置加载（ConfigManager + 单元测试）
任务 2：日志系统初始化（Logger + 单元测试）
任务 3：网络连接建立（NetworkManager + 集成测试）
任务 4：数据解析引擎（Parser + 单元测试）
```

### 第四步：编写任务卡片

每个任务遵循以下结构：

```markdown
## 任务 [N]：[简短描述]

**描述：** 一段话说明任务要完成什么。

**验收标准：**
- [ ] [可测试的具体条件]
- [ ] [可测试的具体条件]

**验证方式：**
- [ ] 编译通过：`cmake --build build`
- [ ] 测试通过：`cmake --build build --target test -- [test_filter]`
- [ ] 无 Sanitizer 报错

**依赖：** [依赖的任务编号，或 "无"]
**涉及文件：**
- `src/module/class.cpp`
- `include/module/class.h`
- `tests/module/class_test.cpp`

**预估规模：** [S: 1-2 文件 | M: 3-5 文件 | L: 5-8 文件]
```

### 第五步：排序和检查点

任务按以下原则排序：
1. 基础组件优先
2. 每完成 2-3 个任务设置一个验证检查点
3. 高风险任务排前面（早失败）

## 任务规模指南

| 规模 | 文件数 | 典型场景 |
|------|--------|----------|
| **S** | 1-2 | 添加一个工具函数、修一个 Bug |
| **M** | 3-5 | 实现一个类及其测试 |
| **L** | 5-8 | 实现一个功能模块 |
| **XL** | 8+ | **太大，需要进一步拆分** |

如果一个任务是 L 或 XL，应该拆成更小的子任务。Agent 在 S 和 M 任务上表现最好。

## C++ 特殊注意事项

- **编译时间**：大量模板改动会导致全量重编译，一个任务不应触发超过 10 个翻译单元重编译
- **头文件改动**：修改 `.h` 文件的成本远高于修改 `.cpp`。尽量将实现放在 `.cpp` 中
- **前向声明**：能用前向声明代替 `#include` 的地方要标注（减少编译依赖）
- **pimpl 模式**：如果涉及大幅度接口变更，考虑是否需要 pimpl 来隔离

## 常见借口

| 借口 | 现实 |
|------|------|
| "拆任务浪费时间" | 10 分钟拆解省下数小时的混乱重写 |
| "任务很明显不需要写" | 写下来才能发现隐藏的依赖和遗漏的边界条件 |
| "全在心里" | 写下来的计划才能跨越编程会话的间隔 |

## 红旗信号

- 没有书面任务列表就开始实现
- 任务的验收标准模糊不清
- 没有验证步骤
- 全部任务都是 XL 级别
- 依赖顺序混乱

## 验证

- [ ] 每个任务都有明确的验收标准
- [ ] 每个任务都有验证方式
- [ ] 依赖关系正确排列
- [ ] 没有任务超过 8 个文件
- [ ] 高风险任务排在前面
- [ ] 计划已自动保存到 `docs/plans/`
