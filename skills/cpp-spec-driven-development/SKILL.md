# C++ 规格驱动开发

## 概述

在写任何 C++ 代码之前，先写一份结构化规格说明。规格定义了要构建什么、为什么、技术约束以及"完成"的标准。没有规格的代码就是在猜谜。

**自动化策略：** Agent 将自动生成完整的规格文档，仅在以下情况中断你：需要的第三方库选择、公共 API 设计决策、架构取舍（权衡）。

## 何时使用

- 新功能影响超过 3 个文件
- 涉及公共接口变更（头文件修改）
- 需要引入新的第三方库
- 预计工作量超过 2 小时
- 涉及多线程或内存管理策略选择

**不使用的情况：** 单文件 Bug 修复、拼写修正、单行配置变更。

## 四阶段门控流程

```
编写规格 ──→ 技术方案 ──→ 任务拆解 ──→ 开始实现
   │            │            │            │
   ▼            ▼            ▼            ▼
 自动生成    自动生成    自动生成    自动执行
 仅关键决策   仅关键决策   无需确认    编译+测试
 才确认      才确认                  自动验证
```

### 阶段 1：编写规格

**自动生成规格文档，覆盖这六个核心区域：**

#### 1.1 目标
- 我们要构建什么？为什么？
- 用户是谁？使用场景是什么？
- 成功的标准是什么（可测量的指标）？

#### 1.2 构建命令
```bash
# 自动从现有项目检测，或设置默认值
cmake --build build --config Release    # 编译
cmake --build build --target test       # 测试
cmake --build build --target format     # 格式化（clang-format）
```

#### 1.3 项目结构
```
src/               → 源代码
include/           → 公共头文件
tests/             → 单元测试（Google Test / Catch2）
benchmarks/        → 性能基准测试
third_party/       → 第三方依赖
docs/              → 文档
```

#### 1.4 代码风格（自动从项目现有代码推断）
```cpp
// Agent 自动读取项目现有代码，提取命名规范，例如：
// - 类型名：PascalCase（class ConfigManager）
// - 函数名：camelCase（void loadConfig()）
// - 成员变量：m_ 前缀（int m_count）
// - 常量：k 前缀（constexpr int kMaxConnections = 32）
// - 命名空间：company::project::module
//
// ===== 注释规范（必须） =====
// - 所有注释用中文编写，详细说明"为什么"而非仅"是什么"
// - 每个类/结构体需要注释说明其职责和生命周期
// - 每个公开函数需要注释说明功能、参数含义、返回值、可能抛出的异常
// - 关键算法段落需要注释解释设计决策
// - 每个成员变量需要注释说明含义、约束和默认值
//
// ===== 编码规范（必须） =====
// - 源码文件保存为 UTF-8 with BOM（MSVC 正确识别中文的必需条件）
// - 代码中需要处理用户可见文本时，使用宽字符 std::wstring
// - Windows API 调用统一使用 W 后缀版本（如 CreateFileW）
// - 控制台输出中文前调用 SetConsoleOutputCP(CP_UTF8)
```

#### 1.5 测试策略
- 测试框架：Google Test（默认，除非项目用 Catch2/doctest）
- 单元测试：每个公开函数至少一个测试
- 集成测试：跨模块交互
- 覆盖率目标：核心逻辑 ≥ 80%
- 内存安全：所有测试在 AddressSanitizer 下运行

#### 1.6 UI 实现策略

所有 UI 界面实现遵循原生优先原则：

1. **Win32 原生 API** 是最优先选择 —— `CreateWindowEx`、窗口类注册、消息循环
2. **Direct2D** 用于自定义绘制 —— 高性能 2D 渲染，与原生 API 天然集成
3. **轻量封装库** —— 可将常用控件/窗口模式封装为内部库，但保持零外部 UI 依赖
4. **不主动引入大型 UI 框架** —— 除非项目已使用 Qt 等框架，否则不引入

#### 1.7 边界规则

**自动执行（无需确认）：**
- 编译、测试、格式化
- 添加源文件/头文件
- 添加测试用例
- 修改 `.cpp` 实现文件
- `#include` 标准库/项目内头文件
- 在注释中说明编码策略（UTF-8 BOM / 宽字符使用）

**仅关键决策确认：**
- 引入新的第三方库（包括 UI 框架）
- 修改公共头文件中的接口签名
- 添加新的编译目标

**禁止：**
- 提交 secrets/密钥
- 修改第三方库代码
- 删除项目目录外的文件

### 阶段 2：技术方案

根据规格，自动生成技术实现方案：

1. 识别主要组件及其依赖关系
2. 确定实现顺序
3. 标注风险和缓解措施
4. 定义统一的错误处理模式（返回码 vs 异常 vs `std::expected`）

```markdown
## 组件依赖图
ConfigManager → Logger → NetworkManager → UserSession
```

### 阶段 3：任务拆解

将方案拆解为离散的可实现任务（详见 `cpp-planning-and-task-breakdown`）。

### 阶段 4：开始实现

按照 `cpp-incremental-implementation` 和 `cpp-test-driven-development` 执行。

## 简洁规格原则

非所有任务都需要完整的 6 页规格。根据复杂度调整：

- **小改动**（1-2 文件）：规格可以只有 5 行 —— 目标 + 验收标准
- **中改动**（3-5 文件）：简化的 6 区域规格
- **大改动**（5+ 文件）：完整规格文档

## 常见借口

| 借口 | 现实 |
|------|------|
| "这个很简单，不用写规格" | 简单任务不需要长规格，但至少要写验收标准。两行规格也可以 |
| "写完代码再补文档" | 那是文档不是规格。规格的价值在于编码前就理清思路 |
| "写规格太慢" | 15 分钟的规格可以防止数小时的重写 |
| "需求反正会变" | 正因为会变，才需要规格来追踪变更。过时的规格也好过没有规格 |

## 红旗信号

- 没写任何需求就开始写代码
- 不清楚"完成"的定义就开始实现
- 做了架构决策但没记录理由

## 验证

在进入实现阶段前确认：

- [ ] 规格覆盖了全部六个核心区域（至少简化版）
- [ ] 构建命令已验证可执行
- [ ] 代码风格与项目一致
- [ ] 边界规则明确（自动/确认/禁止三档）
- [ ] 规格已自动保存到 `docs/specs/`
